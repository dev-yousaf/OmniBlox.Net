Settings CRUD, Global Query Filters & Product Management

This lesson builds on Lesson 3's multi-tenant foundation. We'll create 6 settings features (Brands, Categories, SubCategories, Units, Warranties, VariantAttributes) using a reusable CQRS pattern, then extend it with Products — the first full-featured business entity with stock management, bulk operations, and import/export.

Part 1 — Multi-Tenant Architecture (CompanyId Pattern)

Every OmniBlox request belongs to a company (tenant). The multi-tenant architecture uses global query filters to automatically scope every database query to the current company. No manual .Where(p => p.CompanyId == ...) required.

How every request is automatically scoped to one company
When a user authenticates, the JWT contains a companyId claim. The CurrentUserService reads this claim and exposes it as CompanyId. When AppDbContext builds its model, it inspects every entity type: if the entity implements ITenantEntity (which has a CompanyId property), it attaches a global query filter: e => e.CompanyId == currentCompanyId. Every LINQ query — ToListAsync, FirstOrDefaultAsync, CountAsync — automatically includes this WHERE clause.
File chain: ITenantEntity.cs → AppDbContext.cs → CurrentUserService.cs → handlers

Interface, not base class: Entities already inherit from BaseEntity. C# doesn't allow multiple inheritance, so an interface is the only way to add tenant behavior across the hierarchy.

2. AppDbContext — query filters + auto-set CompanyId

src/OmniBlox.Infrastructure/Persistence/AppDbContext.cs
using System.Linq.Expressions;
using Microsoft.EntityFrameworkCore;
using Microsoft.EntityFrameworkCore.ChangeTracking;
using OmniBlox.Application.Common.Interfaces;
using OmniBlox.Domain.Entities;
using OmniBlox.Domain.Interfaces;

namespace OmniBlox.Infrastructure.Persistence;

public class AppDbContext : DbContext, IApplicationDbContext
{
    private readonly ICurrentUserService _currentUser;

    public AppDbContext(
        DbContextOptions<AppDbContext> options,
        ICurrentUserService currentUser)
        : base(options)
    {
        _currentUser = currentUser;
    }

    public DbSet<Company> Companies => Set<Company>();
    public DbSet<User> Users => Set<User>();
    public DbSet<Product> Products => Set<Product>();
    public DbSet<Brand> Brands => Set<Brand>();
    public DbSet<ProductCategory> ProductCategories => Set<ProductCategory>();
    public DbSet<SubCategory> SubCategories => Set<SubCategory>();
    public DbSet<Unit> Units => Set<Unit>();
    public DbSet<Warranty> Warranties => Set<Warranty>();
    public DbSet<VariantAttribute> VariantAttributes => Set<VariantAttribute>();

    protected override void OnModelCreating(ModelBuilder builder)
    {
        base.OnModelCreating(builder);
        builder.ApplyConfigurationsFromAssembly(typeof(AppDbContext).Assembly);

        foreach (var entityType in builder.Model.GetEntityTypes())
        {
            if (typeof(ITenantEntity).IsAssignableFrom(entityType.ClrType))
            {
                var parameter = Expression.Parameter(entityType.ClrType, "e");
                var property = Expression.Property(parameter, nameof(ITenantEntity.CompanyId));
                var companyId = Expression.Constant(_currentUser.CompanyId);
                var body = Expression.Equal(property, companyId);
                var lambda = Expression.Lambda(body, parameter);

                entityType.SetQueryFilter(lambda);
            }
        }
    }

    public override async Task<int> SaveChangesAsync(CancellationToken ct = default)
    {
        foreach (var entry in ChangeTracker.Entries<ITenantEntity>())
        {
            if (entry.State == EntityState.Added && entry.Entity.CompanyId == Guid.Empty)
            {
                entry.Entity.CompanyId = _currentUser.CompanyId;
            }
        }

        return await base.SaveChangesAsync(ct);
    }
}
Expression trees: The HasQueryFilter method accepts a Lambda expression. Since we don't know entity types at compile time, we build the expression at runtime using Expression.Parameter, Expression.Property, Expression.Constant, Expression.Equal, and Expression.Lambda.
User does NOT implement ITenantEntity: Login needs to look up users across all companies. User access is controlled manually via JWT claims rather than a global filter.

public record CreateBrandCommand : IRequest<BrandDto>
{
    public string Name { get; init; } = string.Empty;
    public string? Description { get; init; }

}

public class CreateBrandCommandHandler : IRequestHandler<CreateBrandCommand, BrandDto>
{
    private readonly IApplicationDbContext _context;
    private readonly ICurrentUserService _currentUser;
    public CreateBrandCommandHandler(IApplicationDbContext context, ICurrentUserService currentUser)
    {
        _context = context;
        _currentUser = currentUser;
    }

    public async Task<BrandDto> Handle(CreateBrandCommand request, CancellationToken ct)
    {
        var slug = request.Name.ToLowerInvariant().Replace(" ", "-");
        var slugExists = await _context.Brands.AnyAsync(b => b.Slug == slug, ct);
        if (slugExists) throw new ConflictException($"Brand '{request.Name}' already exists.");

        var entity = new Brand
        {
            Name = request.Name, Slug = slug,
            Description = request.Description, ImageUrl = request.ImageUrl,
            CompanyId = _currentUser.CompanyId,
        };
        _context.Brands.Add(entity);
        await _context.SaveChangesAsync(ct);
        return BrandDto.FromEntity(entity);
    }
}

    {
        _context = context;
        _currentUser = currentUser;

    public async Task<SubCategoryDto> Handle(CreateSubCategoryCommand request, CancellationToken ct)
    {
        var catExists = await _context.ProductCategories.AnyAsync(x => x.Id == request.CategoryId, ct);

        var entity = new SubCategory
        {
            CategoryId = request.CategoryId, Name = request.Name, Slug = slug,
            Code = request.Code, ImageUrl = request.ImageUrl, Description = request.Description,
            CompanyId = _currentUser.CompanyId,

            product.UpdatedAt = DateTime.UtcNow;
            updated++;

        return new BulkOperationResponse { Updated = updated, Errors = errors };
    }

  description?: string;
  imageUrl?: string;
}

export interface UpdateBrandDto {
  name?: string;
  description?: string;
  imageUrl?: string;
}

export function useBrandsApi() {
  const { get, post, put, del } = useAuthenticatedApi();

  const getBrands = useCallback(async () =>
    get<Brand[]>("/brands"), []);

  const getBrand = useCallback(async (id: string) =>
    get<Brand>(`/brands/${id}`), []);

  const createBrand = useCallback(async (data: CreateBrandDto) =>
    post<Brand>("/brands", data), []);

  const updateBrand = useCallback(async (id: string, data: UpdateBrandDto) =>
    put<Brand>(`/brands/${id}`, data), []);

  const deleteBrand = useCallback(async (id: string) =>
    del(`/brands/${id}`), []);

  const bulkDeleteBrands = useCallback(async (ids: string[]) =>
    post<BulkDeleteResponse>("/brands/bulk-delete", { ids }), []);

  return { getBrands, getBrand, createBrand, updateBrand, deleteBrand, bulkDeleteBrands };
}
Hook pattern: Every feature hook follows the same template — 4 CRUD methods + bulk delete. Only the entity type, DTO types, and endpoint path change. All hooks use useCallback for referential stability. The base useAuthenticatedApi() handles auth, JSON headers, and errors.

Part 10 — Key Architectural Takeaways

What makes this architecture scale
1. Consistent pattern removes cognitive load
Every settings feature has the same 7 files in the same structure. Once you've seen Brands, you've seen all 6. The only differences are domain-specific fields (Duration on Warranty, Values on VariantAttribute, CategoryId on SubCategory).

2. CompanyId is invisible
Global query filters + SaveChangesAsync override mean no handler ever reads or writes CompanyId. The code is declarative — "create brand with this name" — and infrastructure handles tenancy.

3. Zero DI registration per feature
MediatR discovers handlers, FluentValidation discovers validators, EF Core discovers configurations — all by scanning assemblies. Adding a new feature means adding files, not modifying DI configuration.

4. Partial update via nullable merge
All update commands use nullable fields. If the client sends { "name": "New" }, only Name (and Slug) changes. No need for full-object PUT or complex JSON Patch.

5. Shared patterns for common operations
BulkDeleteResponse + FailedItem are used by all bulk delete commands. BulkOperationResponse + BulkOperationError are used by bulk status updates. Consistent error shapes mean the frontend handles errors the same way everywhere.

6. Export/import is a query/command
No separate service layer. Export is a Query that returns byte[]. Import is a Command that accepts a file. MediatR handles the dispatch, the controller returns the result. Testable without HTTP.

7. One controller shape for all settings
BrandsController is the template. CategoriesController, UnitsController, etc. are copy-paste with different types. The ProductsController extends the pattern with extra operations but keeps the same 5 CRUD endpoints.
