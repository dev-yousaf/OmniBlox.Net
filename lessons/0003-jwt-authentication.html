JWT Authentication & the Auth Controller

In NestJS you used Better Auth with cookie-based sessions. In ASP.NET Core we'll use hybrid auth — the same JWT works via both Authorization: Bearer <token> header (for API clients) and an HttpOnly cookie (for browser clients). On login/signup, the server returns the JWT in the response body and sets an HttpOnly cookie. The JWT Bearer middleware reads from either source.

Why hybrid? Browser clients benefit from HttpOnly cookies (they're sent automatically, no JS token management, protected against XSS). API/mobile clients use the Bearer header. Both use the same token, same validation, no duplication.

The Big Picture — Auth Flow

How a request flows through the system
When the client sends POST /auth/login, the request enters the ASP.NET middleware pipeline, hits AuthController, which sends a MediatR command. The handler queries the database via IApplicationDbContext, verifies the password against the BCrypt hash, calls IJwtService.GenerateToken() to produce a signed JWT, and returns the token + user data.

On subsequent requests like GET /auth/me, the client includes the JWT in the Authorization: Bearer <token> header. The JWT Bearer middleware intercepts the request before it reaches the controller — it validates the signature, checks expiry, reads the claims, and populates HttpContext.User. The handler reads the user's identity from ICurrentUserService (which wraps HttpContext.User claims) instead of from the request body. The controller never touches HttpContext directly — it stays clean and testable.

using System.Net;
using System.Text.Json;
using OmniBlox.Shared.Exceptions;

namespace OmniBlox.Api.Middleware;

public class ExceptionHandlingMiddleware
{
    private readonly RequestDelegate _next;

    public ExceptionHandlingMiddleware(RequestDelegate next)
    {
        _next = next;
    }

    public async Task InvokeAsync(HttpContext context)
    {
        try
        {
            await _next(context);
        }
        catch (Exception ex)
        {
            await HandleExceptionAsync(context, ex);
        }
    }

    private static async Task HandleExceptionAsync(
        HttpContext context, Exception ex)
    {
        var (statusCode, message) = ex switch
        {
            UnauthorizedException
                => (HttpStatusCode.Unauthorized, ex.Message),
            NotFoundException
                => (HttpStatusCode.NotFound, ex.Message),
            ConflictException
                => (HttpStatusCode.Conflict, ex.Message),
            FluentValidation.ValidationException validationEx
                => (HttpStatusCode.BadRequest,
                    validationEx.Errors.First().ErrorMessage),
            _ => (HttpStatusCode.InternalServerError,
                    "An error occurred.")
        };

        context.Response.ContentType = "application/json";
        context.Response.StatusCode = (int)statusCode;

        var result = JsonSerializer.Serialize(new
        {
            message,
            statusCode = (int)statusCode
        });

        await context.Response.WriteAsync(result);
    }
}
Why message not error? The Next.js API client (src/OmniBlox.Web/lib/api.ts) expects {"message": "..."} — matching the field name makes error messages display correctly in the frontend without fallback handling.

Syntax breakdown — Middleware:
var (statusCode, message) = ex switch { ... } — C#'s switch expression with tuple deconstruction. Checks the exception type and returns a tuple. _ => is the default case (500).
Note the FluentValidation.ValidationException case — sends validation errors as 400 instead of 500. This is easy to miss but important.
What happens when a handler throws an exception
When a handler does throw new UnauthorizedException("Invalid email or password."), the exception propagates up through MediatR's pipeline, out of the controller action, and into the ASP.NET middleware pipeline. ExceptionHandlingMiddleware wraps everything below it in try/catch — so any unhandled exception from any controller action is caught here. The ex switch { ... } expression pattern-matches the exception type: UnauthorizedException → HTTP 401, NotFoundException → 404, ConflictException → 409, FluentValidation.ValidationException → 400, everything else → 500. The middleware writes a JSON response with {"message": "...", "statusCode": 401} directly to HttpContext.Response and short-circuits the pipeline — no controller code runs after the throw.
File chain: LoginCommandHandler.cs / SignupCommandHandler.cs → AuthController.cs → ExceptionHandlingMiddleware.cs

src/OmniBlox.Infrastructure/Auth/JwtService.cs
using System.IdentityModel.Tokens.Jwt;
using System.Security.Claims;
using System.Text;
using Microsoft.Extensions.Options;
using Microsoft.IdentityModel.Tokens;
using OmniBlox.Application.Common.Interfaces;
using OmniBlox.Domain.Entities;

namespace OmniBlox.Infrastructure.Auth;

public class JwtSettings
{
    public string Secret { get; set; } = string.Empty;
    public string Issuer { get; set; } = string.Empty;
    public string Audience { get; set; } = string.Empty;
    public int ExpiryDays { get; set; } = 7;
}

public class JwtService : IJwtService
{
    private readonly JwtSettings _settings;

    public JwtService(IOptions<JwtSettings> settings)
    {
        _settings = settings.Value;
    }

    public string GenerateToken(User user, Company company)
    {
        var claims = new[]
        {
            new Claim(ClaimTypes.NameIdentifier,
                user.Id.ToString()),
            new Claim(ClaimTypes.Email, user.Email),
            new Claim(ClaimTypes.Role,
                user.Role.ToString()),
            new Claim(ClaimTypes.Name, user.Name),
            new Claim("companyId",
                company.Id.ToString()),
            new Claim("companyName", company.Name),
        };

        var key = new SymmetricSecurityKey(
            Encoding.UTF8.GetBytes(_settings.Secret));

        var credentials = new SigningCredentials(
            key, SecurityAlgorithms.HmacSha256);

        var token = new JwtSecurityToken(
            issuer: _settings.Issuer,
            audience: _settings.Audience,
            claims: claims,
            expires: DateTime.UtcNow
                .AddDays(_settings.ExpiryDays),
            signingCredentials: credentials);

        return new JwtSecurityTokenHandler()
            .WriteToken(token);
    }
}
6 claims total: NameIdentifier, Email, Role, Name, companyId, companyName. JwtSettings is populated from appsettings.json "Jwt" via IOptions<T>.

ICurrentUserService (Application) ↔ CurrentUserService (Infrastructure)

src/OmniBlox.Application/Common/Interfaces/ICurrentUserService.cs
namespace OmniBlox.Application.Common.Interfaces;

public interface ICurrentUserService
{
    Guid UserId { get; }
    Guid CompanyId { get; }
    string Role { get; }
    string Email { get; }
}
src/OmniBlox.Infrastructure/Auth/CurrentUserService.cs
using System.Security.Claims;
using Microsoft.AspNetCore.Http;
using OmniBlox.Application.Common.Interfaces;

namespace OmniBlox.Infrastructure.Auth;

public class CurrentUserService : ICurrentUserService
{
    private readonly IHttpContextAccessor _httpContextAccessor;

    public CurrentUserService(
        IHttpContextAccessor httpContextAccessor)
    {
        _httpContextAccessor = httpContextAccessor;
    }

    public Guid UserId
    {
        get
        {
            var value = _httpContextAccessor
                .HttpContext?.User
                .FindFirstValue(
                    ClaimTypes.NameIdentifier);

            return value is not null
                ? Guid.Parse(value)
                : throw new UnauthorizedAccessException(
                    "User not authenticated.");
        }
    }

    public Guid CompanyId
    {
        get
        {
            var value = _httpContextAccessor
                .HttpContext?.User
                .FindFirstValue("companyId");

            return value is not null
                ? Guid.Parse(value)
                : Guid.Empty;
        }
    }

    public string Role =>
        _httpContextAccessor.HttpContext?.User
            .FindFirstValue(ClaimTypes.Role)
            ?? string.Empty;

    public string Email =>
        _httpContextAccessor.HttpContext?.User
            .FindFirstValue(ClaimTypes.Email)
            ?? string.Empty;
}
Interface exposes UserId, CompanyId, Role, Email. Implementation reads them from JWT claims via IHttpContextAccessor — no HttpContext dependency in Application.
How the three interface/implementation pairs work together
1. IApplicationDbContext / AppDbContext: At DI registration time, AddScoped<IApplicationDbContext>(sp => sp.GetRequiredService<AppDbContext>()) ensures both resolve to the same scoped instance. When a handler constructor receives IApplicationDbContext, DI hands it the single AppDbContext for this request. The handler queries _context.Users and _context.Companies against that one instance.

2. IJwtService / JwtService: When a handler calls _jwtService.GenerateToken(user, company), JwtService reads JwtSettings (populated from appsettings.json via IOptions<T>), builds a SymmetricSecurityKey from the secret, creates 6 claims (userId, email, role, name, companyId, companyName), signs the token with HMAC-SHA256, and serializes it to a compact JWT string. The token embeds the issuer, audience, and expiry — all of which the JWT middleware will validate on subsequent requests.

3. ICurrentUserService / CurrentUserService: On an authenticated request, after JWT middleware has set HttpContext.User, CurrentUserService reads claims via IHttpContextAccessor. UserId reads ClaimTypes.NameIdentifier and parses it to Guid. CompanyId reads the custom "companyId" claim. If there's no authenticated user, it throws. This service is the only code that touches HttpContext — handlers stay completely HTTP-agnostic.
File chain: DependencyInjection.cs → AppDbContext.cs → JwtService.cs → CurrentUserService.cs

using Microsoft.EntityFrameworkCore;

public record LoginCommand : IRequest<AuthResponse>
{

public class LoginCommandHandler : IRequestHandler<LoginCommand, AuthResponse>
{
    private readonly IApplicationDbContext _context;
    private readonly IJwtService _jwtService;

    public LoginCommandHandler(IApplicationDbContext context, IJwtService jwtService)
    {
        _context = context;
        _jwtService = jwtService;
    }

    public async Task<AuthResponse> Handle(LoginCommand request, CancellationToken ct)
    {
        var user = await _context.Users
            .Include(u => u.Company)
            .FirstOrDefaultAsync(u => u.Email == request.Email, ct);

        if (user is null)
            throw new UnauthorizedException("Invalid email or password.");

        if (!BCrypt.Net.BCrypt.Verify(request.Password, user.PasswordHash))
            throw new UnauthorizedException("Invalid email or password.");

        if (user.Status != UserStatus.ACTIVE)
            throw new UnauthorizedException("Account is not active.");

        user.LastLoginAt = DateTime.UtcNow;
        await _context.SaveChangesAsync(ct);

        var token = _jwtService.GenerateToken(user, user.Company);

        var companyDto = new CompanyDto
        {
            Id = user.Company.Id,
            Name = user.Company.Name,
            WorkspaceUrl = user.Company.WorkspaceUrl,
            Industry = user.Company.Industry,
            OtherIndustry = user.Company.OtherIndustry,
            Country = user.Company.Country,
        };

        return new AuthResponse
        {
            Token = token,
            User = new UserDto
            {
                Id = user.Id,
                Email = user.Email,
                Name = user.Name,
                Role = user.Role.ToString(),
                CompanyId = user.CompanyId,
                Company = companyDto,
            },
            Company = companyDto,
        };
    }
}

        await _context.SaveChangesAsync(ct);

        var companyDto = new CompanyDto
        {
            Id = company.Id,
            Name = company.Name,
            WorkspaceUrl = company.WorkspaceUrl,
            Industry = company.Industry,
            OtherIndustry = company.OtherIndustry,
            Country = company.Country,

            Token = token,
            User = new UserDto
            {
                Id = user.Id,
                Email = user.Email,
                Name = user.Name,
                Role = user.Role.ToString(),

    }

    {
        var value = user.FindFirstValue("companyId");

Syntax: this ClaimsPrincipal user — extension method. Adds GetUserId() to any ClaimsPrincipal. Called like User.GetUserId().
How extension methods work at runtime
These extension methods add GetUserId(), GetCompanyId(), and GetRole() to any ClaimsPrincipal object. Inside a controller action, you call them like User.GetUserId() as if they were built-in methods. Each method reads a specific claim by type: ClaimTypes.NameIdentifier, "companyId", or ClaimTypes.Role. If a claim is missing, Guid.Parse() throws (for ID methods) or string.Empty is returned (for Role). These are convenience helpers — the same logic exists in CurrentUserService for the Application layer, but controllers can use these without needing the service injected.
File chain: ClaimsPrincipalExtensions.cs → used in AuthController.cs

        IConfiguration configuration)
    {
        services.AddDbContext<AppDbContext>(options =>
            options.UseNpgsql(
                configuration.GetConnectionString("Postgres"),
                b => b.MigrationsAssembly(typeof(AppDbContext).Assembly.FullName)));

        services.AddScoped<IApplicationDbContext>(sp =>
            sp.GetRequiredService<AppDbContext>());

        services.AddScoped<ICurrentUserService, CurrentUserService>();

        services.Configure<JwtSettings>(
            configuration.GetSection("Jwt"));

        services.AddScoped<IJwtService, JwtService>();

        return services;
    }
}
Why the factory pattern for IApplicationDbContext? AddScoped<IApplicationDbContext>(sp => sp.GetRequiredService<AppDbContext>()) ensures that both AppDbContext and IApplicationDbContext resolve to the same instance. Otherwise you'd get two separate DB connections per request.
How DI registration affects the running app
When Program.cs calls builder.Services.AddApplication(), the extension method calls AddMediatR(cfg => cfg.RegisterServicesFromAssembly(...)) which scans the Application assembly for all IRequestHandler<,> implementations (LoginCommandHandler, SignupCommandHandler, GetCurrentUserQueryHandler) and registers them. AddValidatorsFromAssembly does the same for AbstractValidator<T> classes. No individual handler or validator needs manual registration — they're all auto-discovered.

When AddInfrastructure(...) runs, it registers AppDbContext (scoped), IApplicationDbContext → same scoped AppDbContext, ICurrentUserService → CurrentUserService (scoped), IJwtService → JwtService (scoped), and binds JwtSettings from config. The factory for IApplicationDbContext guarantees that both the AppDbContext registration and the IApplicationDbContext registration resolve to the exact same instance per request — no duplicate connections. When a handler constructor asks for IApplicationDbContext, DI hands it the same AppDbContext that was created for this request.
File chain: Program.cs → DependencyInjection.cs (Application) → DependencyInjection.cs (Infrastructure) → AppDbContext.cs → JwtService.cs → CurrentUserService.cs

Registration order (from top to bottom):
1. AddCors() — allows localhost:3000 (Next.js frontend) with credentials.
2. AddHttpContextAccessor() — enables IHttpContextAccessor for CurrentUserService.
3. AddApplication() — MediatR + FluentValidation.
4. AddAuthentication().AddJwtBearer() — JWT validation + cookie event (checks access_token cookie if no Bearer header).
5. AddAuthorization() — enables [Authorize] / [AllowAnonymous].
6. AddInfrastructure() — DbContext + JwtService + CurrentUserService.

Middleware order (runs top to bottom on every request):
ExceptionHandlingMiddleware → HttpsRedirection → Cors → Authentication → Authorization → MapControllers.
If Authentication came after Authorization, [Authorize] would fail because no user is set yet.
How a complete authenticated request flows through the pipeline
At app startup, Program.cs builds the service collection and middleware pipeline in a critical order.

Service registration order:
0. AddCors() — allows the Next.js frontend (localhost:3000) to make cross-origin requests with cookies (AllowCredentials).
1. AddHttpContextAccessor() — registers IHttpContextAccessor singleton so CurrentUserService can access HttpContext from outside controllers.
2. AddApplication() — registers MediatR handlers + FluentValidation validators via assembly scanning.
3. AddAuthentication().AddJwtBearer() — registers the JWT Bearer handler. The TokenValidationParameters configure what to validate (issuer, audience, lifetime, signing key) — all read from appsettings.json Jwt section. The OnMessageReceived event checks for a access_token cookie first; if present, it uses that as the token instead of the Authorization header. This handler inspects every incoming request for a JWT (from either source), validates it, and sets HttpContext.User with the decoded claims.
4. AddAuthorization() — enables [Authorize] / [AllowAnonymous] infrastructure.
5. AddInfrastructure() — registers DbContext, JwtService, CurrentUserService.

Middleware execution order (runs top to bottom on every request):
1. ExceptionHandlingMiddleware — outermost catch-all.
2. UseHttpsRedirection — HTTP → HTTPS.
3. UseCors() — applies CORS policy (allows localhost:3000 with credentials).
4. UseAuthentication() — JWT Bearer middleware reads the token, validates it, sets HttpContext.User.
5. UseAuthorization() — checks [Authorize] / [AllowAnonymous]; returns 401/403 if unauthorized.
6. MapControllers() — routes to the matching controller action.

When POST /auth/signup arrives with no token: Auth middleware finds no token (no user set), but [AllowAnonymous] means Authorization lets it through → controller runs normally and sets the HttpOnly cookie on the response. When GET /auth/me arrives with a valid cookie or Bearer token: The OnMessageReceived event reads the token from the cookie (if no Bearer header is present), Auth middleware validates the JWT and sets HttpContext.User → Authorization sees [Authorize] and allows → controller reads the user identity from ICurrentUserService. The cookie is sent automatically by the browser — the frontend never needs to manage tokens in JavaScript.
File chain: Program.cs → ExceptionHandlingMiddleware.cs → JWT Bearer Middleware → Authorization Middleware → AuthController.cs → respective handler

class ApiClient {
  private baseUrl: string;

  constructor() {
    if (typeof window !== "undefined") {
      this.baseUrl = process.env.NEXT_PUBLIC_API_URL || `${window.location.origin}/api`;
    } else if (process.env.API_URL) {
      this.baseUrl = process.env.API_URL;
    } else {
      this.baseUrl = process.env.NEXT_PUBLIC_API_URL || "http://localhost:5112";
    }
  }

  private async request<T>(endpoint: string, options: RequestInit = {}): Promise<T> {
    const url = `${this.baseUrl}${endpoint}`;
    const headers: Record<string, string> = {
      "Content-Type": "application/json",
      ...(options.headers as Record<string, string>),
    };

    const response = await fetch(url, {
      ...options, headers, credentials: "include",
    });

    if (!response.ok) {
      const rawText = await response.text().catch(() => "");
      let errorData: any = {};
      try { errorData = rawText ? JSON.parse(rawText) : {}; } catch { errorData = { raw: rawText }; }
      const error = {
        message: errorData?.message || `HTTP ${response.status}: ${response.statusText}`,
        statusCode: response.status, details: errorData,
      };
      const logFn = response.status >= 500 ? console.error : console.warn;
      logFn(`API Error: ${error.message} (${options.method || "GET"} ${url}) [${response.status}]`);
      throw error;
    }

    const ct = response.headers.get("content-type");
    return ct?.includes("application/json") ? response.json() : response.text() as any;
  }

  async signup(data) { return this.request("/auth/signup", { method: "POST", body: JSON.stringify(data) }); }
  async login(email, pw) { return this.request("/auth/login", { method: "POST", body: JSON.stringify({ email, password: pw }) }); }
  async logout() { try { await this.request("/auth/logout", { method: "POST" }); } catch {} }
  async getProfile() { return this.request("/auth/me"); }
  async validateToken() { return this.request("/auth/validate"); }
  async get<T>(e) { return this.request(e, { method: "GET" }); }
  async post<T>(e, d?) { return this.request(e, { method: "POST", body: d ? JSON.stringify(d) : undefined }); }
  async put<T>(e, d?) { return this.request(e, { method: "PUT", body: d ? JSON.stringify(d) : undefined }); }
  async patch<T>(e, d?) { return this.request(e, { method: "PATCH", body: d ? JSON.stringify(d) : undefined }); }
  async delete<T>(e) { return this.request(e, { method: "DELETE" }); }
}

14d. Middleware — server-side auth guard (middleware.ts)

Next.js middleware.ts — checks cookies before page renders
const GUEST_PATHS = ["/", "/features", "/pricing", "/about", "/contact",
                      "/login", "/signup", "/forgot-password", "/accept-invitation"];
const OLD_APP_ROUTES = ["/dashboard", "/sales", "/products", ...];

export function middleware(req: NextRequest) {
  const url = req.nextUrl.clone();
  const path = url.pathname;

  const accept = req.headers.get("accept") || "";
  const isHtml = accept.includes("text/html");
  if (!isHtml || path.startsWith("/_next") || path.startsWith("/api")) {
    return NextResponse.next();
  }

  const hasCookie = req.cookies.get("omniblox_logged_in")?.value === "1";
  const workspace = req.cookies.get("omniblox_workspace")?.value;

  if (hasCookie) {
    if (!workspace) { url.pathname = "/login"; return NextResponse.redirect(url); }
    const isGuestPath = GUEST_PATHS.some(g => path === g || path.startsWith(g + "/"));
    const firstSegment = path.split("/").filter(Boolean)[0] || "";
    if (isGuestPath || firstSegment !== workspace) {
      url.pathname = `/${workspace}/dashboard`;
      return NextResponse.redirect(url);
    }
  }

  if (!hasCookie) {
    const isOldRoute = OLD_APP_ROUTES.some(r => path === r || path.startsWith(r + "/"));
    if (isOldRoute) { url.pathname = "/login"; return NextResponse.redirect(url); }
  }

  return NextResponse.next();
}
Cookie-based guard: The middleware only checks omniblox_logged_in — the old better-auth.session_token check was removed as a leftover from NestJS/Better Auth. The .NET API uses only access_token (HttpOnly cookie), which the middleware can't read in JavaScript — hence the companion omniblox_logged_in cookie set by the client.

    try {
      await signup(payload);
      router.push("/login");
    } catch (err: any) {
      const msg = err.message || "An error occurred during signup";

      if (lower.includes("workspace url"))
        setErrors(p => ({ ...p, workspaceUrl: msg }));
      else if (lower.includes("email"))
        setErrors(p => ({ ...p, email: msg }));
      else setError(msg);
    } finally {
      setSubmitting(false);
    }
  }

  return (
    <div className="min-h-screen flex items-center justify-center bg-gradient-to-br
                    from-background via-muted to-background p-4">

              </div>
              <div>

                {errors.cpassword && <p className="text-xs text-red-500">{errors.cpassword}</p>}
              </div>
            </div>
          </section>

          {/* Section 2: Business Details */}
          <section>
            <h2 className="text-xl font-semibold border-b pb-2 mb-6">2. Business Details</h2>
            <div className="grid gap-6 md:grid-cols-2">
              <div>

                {errors.companyName && <p className="text-xs text-red-500">{errors.companyName}</p>}
              </div>
              <div>
                <Label>Workspace URL</Label>
                <Input name="workspaceUrl" placeholder="your-company" />
                {errors.workspaceUrl && <p className="text-xs text-red-500">{errors.workspaceUrl}</p>}
              </div>
              <div>
                <Label>Industry</Label>
                <Select name="industry" onValueChange={setIndustry}>
                  <SelectTrigger><SelectValue placeholder="Select" /></SelectTrigger>

                  </SelectContent>
                </Select>
                {errors.industry && <p className="text-xs text-red-500">{errors.industry}</p>}
              </div>
              {industry === "other" && (
                <div>
                  <Label>Specify Industry</Label>
                  <Input name="otherIndustry" placeholder="e.g., Logistics"
                         value={otherIndustry} onChange={e => setOtherIndustry(e.target.value)} />
                </div>
              )}
              <div>
                <Label>Country</Label>
                <Select name="country">
                  <SelectTrigger><SelectValue placeholder="Select" /></SelectTrigger>
                  <SelectContent>
                    <SelectItem value="us">United States</SelectItem>
                    <SelectItem value="ca">Canada</SelectItem>
                    <SelectItem value="gb">United Kingdom</SelectItem>
                  </SelectContent>
                </Select>
                {errors.country && <p className="text-xs text-red-500">{errors.country}</p>}
              </div>
            </div>
          </section>

          <div className="flex items-start space-x-2">
            <Checkbox id="terms" required />
            <Label htmlFor="terms" className="cursor-pointer text-sm">
              I agree to the <Link href="/terms" className="underline">Terms of Service</Link>
              and <Link href="/privacy" className="underline">Privacy Policy</Link>.
            </Label>
          </div>
          <Button size="lg" type="submit" className="w-full font-semibold" disabled={submitting}>
            {submitting ? "Creating Workspace..." : "Continue"}
          </Button>
        </form>
      </div>
    </div>
  );
}

14j. Environment & .env.local

Configure the API URL — empty = Next.js proxy, set = direct

.env.local (local dev — currently using direct mode)
NEXT_PUBLIC_API_URL=http://localhost:5112
The API client reads process.env.NEXT_PUBLIC_API_URL. When set → direct CORS mode (http://localhost:5112/auth/login). When empty → same-origin proxy (/api/auth/login via next.config.mjs rewrites). The proxy targets http://localhost:5112 to match the API port.
Full auth flow — from signup to dashboard
Signup: User fills the signup form → clicks Continue → form data collected via FormData → POST /auth/signup via API client → API creates Company + User with OWNER role, hashes password, generates JWT, sets access_token HttpOnly cookie → response includes { user, company } → Next.js redirects to /login.

Login: User enters email + password → auth-context.tsx calls api.login(email, password) → API verifies credentials, sets JWT cookie → then calls api.getProfile() immediately to read the cookie and fetch full user profile with company/workspace → sets omniblox_logged_in and omniblox_workspace cookies for middleware → pushes to /{workspace}/dashboard.

Page refresh: AuthProvider.useEffect calls GET /auth/me with the HttpOnly cookie → API returns User → context sets user + workspace cookies → useEffect redirects to /{workspace}/dashboard if needed. If cookie expired, getProfile() throws → user is null → route guard redirects to /login.

Error handling: API errors include {"message": "Description", "statusCode": 401}. The API client throws { message, statusCode }. Login page catches it, checks if message contains "email" (field-level error) or "password" — otherwise displays as a general Alert.
File chain: app/(auth)/login/page.tsx → contexts/auth-context.tsx → lib/api.ts → next.config.mjs proxy or direct → Program.cs → AuthController.cs → LoginCommandHandler.cs → ExceptionHandlingMiddleware.cs
