Lesson 2 · Domain Entities & EF Core Configuration · ~20 minutes

Domain Entities & EF Core Fluent API

In Prisma, you define models in schema.prisma and Prisma generates C# types. In EF Core, you write the C# classes first, then configure how they map to the database using Fluent API. This is called code-first.

Mindset shift: Prisma = schema-first (define DB schema, get C# types). EF Core = code-first (write C# classes, generate DB schema). Your entities are plain C# objects (POCOs) with no EF Core attributes required — though you can use data annotations if you prefer.

1. Enums — value types used by entities

Enums are named constants. They live in Domain/Enums/ and are referenced by the User entity. You'll see them in the User.cs file below.

Enums — referenced by User entity

src/OmniBlox.Domain/Enums/UserRole.cs
namespace OmniBlox.Domain.Enums;

public enum UserRole
{
    OWNER = 0,
    ADMIN = 1,
    MANAGER = 2,
    OBSERVER = 3
}
src/OmniBlox.Domain/Enums/UserStatus.cs
namespace OmniBlox.Domain.Enums;

public enum UserStatus
{
    ACTIVE = 0,
    INVITED = 1,
    SUSPENDED = 2
}
User.cs uses UserRole and UserStatus as property types (scroll down to see)
Syntax breakdown: namespace OmniBlox.Domain.Enums; — file-scoped namespace (C# 10+). Everything in this file belongs to that namespace. Like a folder path for code. No { } needed.
public enum UserRole { ... } — an enum is a set of named constants. The numbers after = are the underlying integer values (stored in DB as these ints).
OWNER = 0 — by convention, 0 is the default value. UserRole without assignment defaults to OWNER.
What happens at runtime
When a User entity is created in memory, Role holds an enum value like UserRole.OWNER. When EF Core saves it to the database, HasConversion<int>() (configured in UserConfiguration) converts it to the integer 0 in the PostgreSQL column. When EF Core reads the row back, it converts the int back to the enum — 0 → UserRole.OWNER. The conversion rules are declared once in the Fluent API config; you never think about it again in code.
File chain: User.cs → UserConfiguration.cs → AppDbContext.cs → PostgreSQL

2. Base class → derived entity (inheritance)

BaseEntity holds common properties. Company and User inherit from it — notice the : BaseEntity syntax.

BaseEntity provides Id, CreatedAt, UpdatedAt → Company inherits them

src/OmniBlox.Domain/Entities/BaseEntity.cs
namespace OmniBlox.Domain.Entities;

public abstract class BaseEntity
{
    public Guid Id { get; set; } = Guid.NewGuid();
    public DateTime CreatedAt { get; set; } = DateTime.UtcNow;
    public DateTime? UpdatedAt { get; set; }
}
src/OmniBlox.Domain/Entities/Company.cs
namespace OmniBlox.Domain.Entities;

public class Company : BaseEntity
{
    public string Name { get; set; } = string.Empty;
    public string Email { get; set; } = string.Empty;
    public string Phone { get; set; } = string.Empty;
    public string? Address { get; set; }
    public string WorkspaceUrl { get; set; } = string.Empty;
    public string? Industry { get; set; }
    public string? Country { get; set; }
    public bool IsActive { get; set; } = true;

    public ICollection<User> Users { get; set; } = [];
}
: BaseEntity → Company gets Id, CreatedAt, UpdatedAt without writing them. WorkspaceUrl → unique slug used in multi-tenant routing (e.g. mycompany.omniblox.com). Industry, Country → optional profile fields collected during signup. ICollection<User> → see User entity below for the other side of this relationship.
Syntax breakdown: abstract class BaseEntity — abstract = can't create new BaseEntity() directly. It's a base for other classes to inherit from.
Guid Id { get; set; } — a property with a getter and setter. Like TypeScript: get Id(): string { ... }. Guid is .NET's UUID.
= Guid.NewGuid() — default value: generates a new UUID when a new entity is created.
DateTime? UpdatedAt — the ? makes it nullable. UpdatedAt will be null until the entity is first updated.
What happens at runtime
When code calls new Company(), the constructor chains up to BaseEntity: Id gets a fresh UUID via Guid.NewGuid(), CreatedAt is set to the current UTC time, and UpdatedAt is null. The abstract keyword prevents anyone from writing new BaseEntity() directly — it only exists as a parent class. When EF Core maps Company to the database, it includes the inherited Id, CreatedAt, and UpdatedAt columns automatically — they're part of the Company table, not a separate table. Every entity that inherits BaseEntity gets these three columns for free.
File chain: Company.cs → BaseEntity.cs → AppDbContext.cs

3. Entity relationship — one Company has many Users

These two files define both sides of the same relationship. Notice Company has a collection of Users; User has a reference to Company + a CompanyId FK.

Navigation properties: Company.Users ↔ User.Company

src/OmniBlox.Domain/Entities/Company.cs
namespace OmniBlox.Domain.Entities;

public class Company : BaseEntity
{
    public string Name { get; set; } = string.Empty;
    public string Email { get; set; } = string.Empty;
    public string Phone { get; set; } = string.Empty;
    public string? Address { get; set; }
    public bool IsActive { get; set; } = true;

    public ICollection<User> Users { get; set; } = [];
}
src/OmniBlox.Domain/Entities/User.cs
using OmniBlox.Domain.Enums;

namespace OmniBlox.Domain.Entities;

public class User : BaseEntity
{
    public string Email { get; set; } = string.Empty;
    public string PasswordHash { get; set; } = string.Empty;
    public string Name { get; set; } = string.Empty;
    public UserRole Role { get; set; } = UserRole.OBSERVER;
    public UserStatus Status { get; set; } = UserStatus.INVITED;
    public DateTime? LastLoginAt { get; set; }

    public Guid CompanyId { get; set; }
    public Company Company { get; set; } = null!;
}
Company.Users (collection nav) ↔ User.CompanyId (FK) + User.Company (reference nav). Role and Status use the enum types from Enums/ above.
Syntax breakdown: using OmniBlox.Domain.Enums; — imports the Enums namespace so we can use UserRole and UserStatus without full path.
public UserRole Role { get; set; } — property of enum type. EF Core stores it as int in DB (0=OWNER, 1=ADMIN etc.).
Guid CompanyId { get; set; } — the foreign key column. Prisma equivalent: companyId String.
Company Company { get; set; } = null!; — navigation property to the parent Company. null! tells the compiler "trust me, this won't be null at runtime" (EF Core sets it). Like TypeScript's ! non-null assertion.
Prisma ↔ EF Core comparison: @id @default(uuid()) → Guid Id + = Guid.NewGuid().
@relation(fields: [companyId], references: [id]) → CompanyId property + Company navigation property.
companyId String → public Guid CompanyId { get; set; }.
What happens at runtime
When you create a new User and assign it a CompanyId, the Company.Users collection stays empty until you manually add the user to it — navigation properties don't auto-sync. But when EF Core loads a Company with .Include(c => c.Users), it runs a JOIN query, creates User objects for each matching row, sets User.Company to the parent Company object, and adds them to Company.Users. The relationship becomes fully traversable from either side: user.Company.Name or company.Users.First().Email. Without .Include(), the navigation properties are null (reference) or empty (collection) — EF Core doesn't lazy-load by default.
File chain: Company.cs → User.cs → UserConfiguration.cs → AppDbContext.cs → PostgreSQL

4. Entity + its Fluent API configuration

Each entity gets a configuration class that tells EF Core how to map it to the database. Compare the entity (left) with its config (right) to see how properties get constraints.

Company entity → CompanyConfiguration (table name, constraints, indexes)

src/OmniBlox.Domain/Entities/Company.cs
namespace OmniBlox.Domain.Entities;

public class Company : BaseEntity
{
    public string Name { get; set; } = string.Empty;
    public string Email { get; set; } = string.Empty;
    public string Phone { get; set; } = string.Empty;
    public string? Address { get; set; }
    public bool IsActive { get; set; } = true;
    public ICollection<User> Users { get; set; } = [];
}
src/OmniBlox.Infrastructure/Persistence/Configurations/CompanyConfiguration.cs
using Microsoft.EntityFrameworkCore;
using Microsoft.EntityFrameworkCore.Metadata.Builders;
using OmniBlox.Domain.Entities;

namespace OmniBlox.Infrastructure.Persistence.Configurations;

public class CompanyConfiguration : IEntityTypeConfiguration<Company>
{
    public void Configure(EntityTypeBuilder<Company> builder)
    {
        builder.ToTable("Companies");

        builder.HasKey(c => c.Id);

        builder.Property(c => c.Name)
            .HasMaxLength(200)
            .IsRequired();

        builder.Property(c => c.Email)
            .HasMaxLength(256)
            .IsRequired();

        builder.Property(c => c.Phone)
            .HasMaxLength(50)
            .IsRequired();

        builder.Property(c => c.Address)
            .HasMaxLength(500);

        builder.Property(c => c.WorkspaceUrl)
            .HasMaxLength(200)
            .IsRequired();

        builder.Property(c => c.Industry)
            .HasMaxLength(100);

        builder.Property(c => c.Country)
            .HasMaxLength(100);

        builder.HasIndex(c => c.Email).IsUnique();
        builder.HasIndex(c => c.WorkspaceUrl).IsUnique();
    }
}
Each builder.Property(...) matches a property in the entity. .HasMaxLength() = VARCHAR(n). .IsRequired() = NOT NULL. .HasIndex(...).IsUnique() = unique constraint.
Syntax breakdown — Fluent API: IEntityTypeConfiguration<Company> — interface with one method Configure. EF Core calls it automatically to learn how Company maps to the DB. Like Prisma's schema block for a model.
builder.ToTable("Companies") — the table name in PostgreSQL. Without this, EF Core would use "Company" (pluralization isn't automatic).
builder.HasKey(c => c.Id) — sets primary key. c => c.Id is a lambda expression — C#'s arrow function. Think (c) => c.Id in TypeScript.
builder.Property(c => c.Name).HasMaxLength(200).IsRequired() — chain of method calls (fluent API): select the column → set max length → make it NOT NULL. Prisma equivalent: name String @db.VarChar(200).
builder.HasIndex(c => c.Email).IsUnique() — unique index on Email column. Prisma equivalent: @unique.

User entity → UserConfiguration (enums as ints, FK relationship)

src/OmniBlox.Domain/Entities/User.cs
using OmniBlox.Domain.Enums;

namespace OmniBlox.Domain.Entities;

public class User : BaseEntity
{
    public string Email { get; set; } = string.Empty;
    public string PasswordHash { get; set; } = string.Empty;
    public string Name { get; set; } = string.Empty;
    public UserRole Role { get; set; } = UserRole.OBSERVER;
    public UserStatus Status { get; set; } = UserStatus.INVITED;
    public DateTime? LastLoginAt { get; set; }
    public Guid CompanyId { get; set; }
    public Company Company { get; set; } = null!;
}
src/OmniBlox.Infrastructure/Persistence/Configurations/UserConfiguration.cs
using Microsoft.EntityFrameworkCore;
using Microsoft.EntityFrameworkCore.Metadata.Builders;
using OmniBlox.Domain.Entities;

namespace OmniBlox.Infrastructure.Persistence.Configurations;

public class UserConfiguration : IEntityTypeConfiguration<User>
{
    public void Configure(EntityTypeBuilder<User> builder)
    {
        builder.ToTable("Users");

        builder.HasKey(u => u.Id);

        builder.Property(u => u.Email)
            .HasMaxLength(256)
            .IsRequired();

        builder.Property(u => u.PasswordHash)
            .HasMaxLength(500)
            .IsRequired();

        builder.Property(u => u.Name)
            .HasMaxLength(200)
            .IsRequired();

        builder.Property(u => u.Role)
            .HasConversion<int>()
            .IsRequired();

        builder.Property(u => u.Status)
            .HasConversion<int>()
            .IsRequired();

        builder.HasIndex(u => u.Email).IsUnique();

        builder.HasOne(u => u.Company)
            .WithMany(c => c.Users)
            .HasForeignKey(u => u.CompanyId)
            .OnDelete(DeleteBehavior.Restrict);
    }
}
Enums: .HasConversion<int>() stores them as integers. Relationship: HasOne → WithMany → HasForeignKey → OnDelete tells EF Core about the FK.
Syntax breakdown — Fluent API (new parts): .HasConversion<int>() — tells EF Core to store the enum as an integer (0, 1, 2...) instead of a string. Default would work too, but explicit is clearer.
builder.HasOne(u => u.Company) — "User has one Company". Starts defining the relationship.
.WithMany(c => c.Users) — "Company has many Users". Completes the relationship definition.
.HasForeignKey(u => u.CompanyId) — "the FK column is User.CompanyId".
.OnDelete(DeleteBehavior.Restrict) — "don't cascade delete". Prevent deleting a Company if it still has users. Prisma equivalent: onDelete: Restrict.
Full chain: HasOne → WithMany → HasForeignKey → OnDelete reads as: "User has one Company which has many Users, joined on User.CompanyId, restrict deletes".
What happens at runtime
At app startup, when the first query runs against AppDbContext, EF Core calls OnModelCreating() once and caches the resulting model. Each configuration class receives a builder — an in-memory metadata builder. Every .HasMaxLength(200), .IsRequired(), .HasConversion<int>() call modifies EF Core's internal model of the database schema. When you later run dotnet ef migrations add, EF Core diffs this model against the previous migration snapshot and generates SQL: CREATE TABLE "Companies" ("Name" VARCHAR(200) NOT NULL, "Email" VARCHAR(256) NOT NULL UNIQUE…). The HasOne → WithMany → HasForeignKey → OnDelete(DeleteBehavior.Restrict) chain generates a foreign key constraint: ALTER TABLE "Users" ADD CONSTRAINT FK_Users_Companies FOREIGN KEY ("CompanyId") REFERENCES "Companies" ("Id") ON DELETE RESTRICT. PostgreSQL will refuse to delete a Company that still has Users.
File chain: AppDbContext.cs → CompanyConfiguration.cs → UserConfiguration.cs → PostgreSQL

5. DbContext — how configurations are discovered

AppDbContext is the Prisma Client equivalent. It scans for all IEntityTypeConfiguration<T> files automatically via ApplyConfigurationsFromAssembly.

AppDbContext scans → discovers CompanyConfiguration & UserConfiguration

src/OmniBlox.Infrastructure/Persistence/AppDbContext.cs
using Microsoft.EntityFrameworkCore;
using OmniBlox.Application.Common.Interfaces;
using OmniBlox.Domain.Entities;

namespace OmniBlox.Infrastructure.Persistence;

public class AppDbContext : DbContext, IApplicationDbContext
{
    public AppDbContext(
        DbContextOptions<AppDbContext> options)
        : base(options) { }

    public DbSet<Company> Companies => Set<Company>();
    public DbSet<User> Users => Set<User>();

    protected override void OnModelCreating(
        ModelBuilder builder)
    {
        base.OnModelCreating(builder);
        builder.ApplyConfigurationsFromAssembly(
            typeof(AppDbContext).Assembly);
    }
}
Discovered automatically (example)
// CompanyConfiguration.cs
public class CompanyConfiguration
    : IEntityTypeConfiguration<Company>
{ ... }

// UserConfiguration.cs
public class UserConfiguration
    : IEntityTypeConfiguration<User>
{ ... }
ApplyConfigurationsFromAssembly scans the Infrastructure assembly for all classes implementing IEntityTypeConfiguration<T> and registers them. Add a new config file? It just works.
Syntax breakdown: public class AppDbContext : DbContext — inherits from EF Core's built-in DbContext class. That's what gives it query/save/migration capabilities.
public AppDbContext(DbContextOptions<AppDbContext> options) : base(options) { } — constructor receives the connection config (DbContextOptions) and passes it to the base class. NestJS equivalent: PrismaClient's constructor.
public DbSet<Company> Companies => Set<Company>(); — DbSet<T> represents a database table. Companies = the "Companies" table. => Set<Company>() is an expression-bodied member (C# 6+) — shorthand for { get { return Set<Company>(); } }. NestJS equivalent: prisma.company.
protected override void OnModelCreating(ModelBuilder builder) — override method called by EF Core when the model is being built (on app startup). This is where all entity configurations are applied.
What happens at runtime
The first time any code queries the database — say _context.Users.FirstOrDefaultAsync() — EF Core calls OnModelCreating() exactly once and caches the resulting model for the lifetime of the application. Inside it, ApplyConfigurationsFromAssembly(typeof(AppDbContext).Assembly) uses reflection to scan the Infrastructure assembly for every class that implements IEntityTypeConfiguration<>, instantiates each one, and calls its Configure() method. This is how CompanyConfiguration and UserConfiguration are discovered — you never register them individually. After OnModelCreating returns, EF Core has a complete in-memory map of all tables, columns, types, keys, indexes, and foreign keys. The DbSet<Company> Companies and DbSet<User> Users properties are the entry points: any LINQ query like _context.Users.Where(u => u.Email == email) is translated to SQL and executed against this model.
File chain: AppDbContext.cs → CompanyConfiguration.cs → UserConfiguration.cs → PostgreSQL

6. Dependency Injection — wiring it all together

How does ASP.NET Core know about our DbContext? Through Dependency Injection. The Infrastructure project registers the DbContext, and Program.cs calls it.

DependencyInjection.cs (registers) → Program.cs (calls)

src/OmniBlox.Infrastructure/DependencyInjection.cs
using Microsoft.EntityFrameworkCore;
using Microsoft.Extensions.Configuration;
using Microsoft.Extensions.DependencyInjection;
using OmniBlox.Application.Common.Interfaces;
using OmniBlox.Infrastructure.Auth;
using OmniBlox.Infrastructure.Persistence;

namespace OmniBlox.Infrastructure;

public static class DependencyInjection
{
    public static IServiceCollection AddInfrastructure(
        this IServiceCollection services,
        IConfiguration configuration)
    {
        services.AddDbContext<AppDbContext>(options =>
            options.UseNpgsql(
                configuration.GetConnectionString("Postgres"),
                b => b.MigrationsAssembly(
                    typeof(AppDbContext).Assembly.FullName)));

        services.AddScoped<IApplicationDbContext>(sp =>
            sp.GetRequiredService<AppDbContext>());

        services.AddScoped<ICurrentUserService, CurrentUserService>();

        services.Configure<JwtSettings>(
            configuration.GetSection("Jwt"));

        services.AddScoped<IJwtService, JwtService>();

        return services;
    }
}
src/OmniBlox.Api/Program.cs
using OmniBlox.Infrastructure;

var builder = WebApplication.CreateBuilder(args);

builder.Services.AddControllers();
builder.Services.AddOpenApi();

builder.Services.AddInfrastructure(builder.Configuration);

var app = builder.Build();

if (app.Environment.IsDevelopment())
{
    app.MapOpenApi();
}

app.UseHttpsRedirection();
app.UseAuthorization();
app.MapControllers();

app.Run();
builder.Services.AddInfrastructure(builder.Configuration) calls the extension method from Infrastructure, which registers AppDbContext with PostgreSQL using the connection string from appsettings.json.
Syntax breakdown — DI Extension Method: public static class DependencyInjection — static class (can't be instantiated, all members are static). Container for related methods.
this IServiceCollection services — the this keyword makes this an extension method. It lets you call services.AddInfrastructure(...) as if it were a built-in method. Like adding a method to an existing class in TypeScript via prototype.
IServiceCollection — ASP.NET Core's DI container. All services (DbContext, repositories, JWT service) are registered here.
services.AddDbContext<AppDbContext>(options => ...) — registers AppDbContext in DI with scoped lifetime (one instance per HTTP request).
options.UseNpgsql(connectionString, b => b.MigrationsAssembly(...)) — tells EF Core to use PostgreSQL. MigrationsAssembly tells EF Core which project contains the migration files (Infrastructure, not Api).
return services; — returns the service collection so calls can be chained.
What happens at runtime
When Program.cs calls builder.Services.AddInfrastructure(builder.Configuration), the extension method runs during app startup. AddDbContext<AppDbContext>() registers AppDbContext with scoped lifetime — one instance per HTTP request. It reads the connection string from ConnectionStrings:Postgres in appsettings.json and passes it to UseNpgsql(). When an HTTP request arrives and a handler constructor requests IApplicationDbContext, ASP.NET's DI container creates a new AppDbContext, injects the PostgreSQL connection options, and hands it to the handler. When the request ends, the container disposes the AppDbContext (closing the database connection). Every request gets a fresh AppDbContext — this is the standard pattern for web applications.
File chain: Program.cs → DependencyInjection.cs → AppDbContext.cs → appsettings.json

7. Connection string (appsettings.json)

The connection string is the .NET equivalent of a .env DATABASE_URL. It lives in the Api project (the startup project) and is read via GetConnectionString("Postgres").

src/OmniBlox.Api/appsettings.json (top section):

{
  "ConnectionStrings": {
    "Postgres": "Host=localhost;Port=5432;Database=Omniblox;Username=postgres;Password=PLACEHOLDER"
  },
  "Logging": { ... },
  "AllowedHosts": "*"
}
Syntax breakdown: "ConnectionStrings" — a special config section. GetConnectionString("Postgres") in the DI code reads ConnectionStrings:Postgres from this file.
Host=localhost;Port=5432;Database=omniblox — standard PostgreSQL connection string format. Adjust if your DB has different credentials.
Important: This file is in the Api project, not Infrastructure. Configuration always lives in the startup project. The Infrastructure project reads it via IConfiguration injected from Api.
Don't have PostgreSQL installed? You can install it locally, or use Docker: docker run -d --name omniblox-db -e POSTGRES_PASSWORD=postgres -p 5432:5432 postgres:16 Then dotnet ef database update will create the tables.
What happens when you run the migration
dotnet ef migrations add InitialCreate compiles the Api project, loads the Infrastructure assembly, discovers AppDbContext, calls OnModelCreating(), diffs the resulting model against an empty database, and generates a migration class in Infrastructure/Persistence/Migrations/ — a C# file with Up() and Down() methods containing the SQL to create/drop the tables. dotnet ef database update then executes the Up() SQL against the live PostgreSQL database — creating the Companies and Users tables with all columns, constraints, indexes, and foreign keys defined in the Fluent API. It also creates an __EFMigrationsHistory table to track which migrations have been applied.
File chain: Program.cs → AppDbContext.cs → CompanyConfiguration.cs → UserConfiguration.cs → Migrations/InitialCreate.cs → PostgreSQL

Verify It Builds

dotnet build

If it compiles, you've successfully created your first entities and wired up EF Core. Next, install the EF Core CLI tool (if you haven't already) and create your first migration:

⚠️ Common gotcha #1 — Design package: The CLI will give you this error if you skip this step:
"Your startup project 'OmniBlox.Api' doesn't reference Microsoft.EntityFrameworkCore.Design."
The Api project (startup project) needs this package even though the migrations live in Infrastructure. Add it first: dotnet add src\OmniBlox.Api\OmniBlox.Api.csproj package Microsoft.EntityFrameworkCore.Design
⚠️ Common gotcha #2 — Database not created yet: The first time you run dotnet ef database update, you'll see a scary-looking error like "Failed executing DbCommand ... SELECT FROM __EFMigrationsHistory". This is normal — it's trying to check the migration history table before the database exists. The command will then create the database and apply the migration. Check the last line: if it says Done., you're good. 
# 0. Install the EF Core CLI tool (one-time)
dotnet tool install --global dotnet-ef

# 1. Add EF Core Design package to the Api project (required by CLI)
dotnet add src\OmniBlox.Api\OmniBlox.Api.csproj package Microsoft.EntityFrameworkCore.Design

# 2. Create the first migration
dotnet ef migrations add InitialCreate ^
    --project src\OmniBlox.Infrastructure ^
    --startup-project src\OmniBlox.Api

# 3. Check the generated migration (will show "Pending" before applying)
dotnet ef migrations list ^
    --project src\OmniBlox.Infrastructure ^
    --startup-project src\OmniBlox.Api

# 4. Apply migrations to the database
dotnet ef database update ^
    --project src\OmniBlox.Infrastructure ^
    --startup-project src\OmniBlox.Api

# 5. If you made a mistake, remove the last migration:
dotnet ef migrations remove ^
    --project src\OmniBlox.Infrastructure ^
    --startup-project src\OmniBlox.Api
Why --project and --startup-project?
--project = where the DbContext is (Infrastructure).
--startup-project = where appsettings.json and Program.cs are (Api).
The ^ character is PowerShell's line continuation (like \ in bash).
Your first migration will contain CreateTable("Companies") and CreateTable("Users") with all the columns and constraints you defined in the Fluent API. Open the generated file in Infrastructure/Persistence/Migrations/ to see the SQL. This is the EF Core equivalent of prisma migrate dev.
