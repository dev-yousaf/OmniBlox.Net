Clean Architecture & Multi-Project Solution

Before writing any code, you need to understand the key syntax elements of a .NET project — the .csproj file and Program.cs — and how they map to concepts you already know from NestJS. Then we'll look at why the solution is split into five projects instead of one. This is the Clean Architecture pattern — it enforces separation of concerns at the compiler level.

NestJS analogy: In NestJS, @Module() decorators create logical boundaries at runtime. In ASP.NET Core, we use project references to create compile-time boundaries. If someone accidentally imports EF Core in the Domain project, dotnet build will fail. That's the whole point.

Anatomy of a .csproj File

Open src/OmniBlox.Api/OmniBlox.Api.csproj. Here's what each line means:

<Project Sdk="Microsoft.NETSdk.Web">    // ← "Web" SDK = this is a web API project
                                       //    "NET.Sdk" = class library

  <PropertyGroup>
    <TargetFramework>net10.0</TargetFramework>   // ← targets .NET 10
    <Nullable>enable</Nullable>                  // ← enables nullable reference types
    <ImplicitUsings>enable</ImplicitUsings>      // ← auto-imports common namespaces
  </PropertyGroup>

  <ItemGroup>
    <PackageReference Include="MediatR" Version="14.2.0" />  // ← NuGet package, like npm package
  </ItemGroup>

  <ItemGroup>
    <ProjectReference Include="..\OmniBlox.Application\...csproj" />  // ← project dependency
  </ItemGroup>

Anatomy of Program.cs

Open src/OmniBlox.Api/Program.cs — this is the entry point, like NestJS's main.ts + AppModule combined:

// 1. BUILD the app (register services, configure DI)
var builder = WebApplication.CreateBuilder(args);
builder.Services.AddControllers();
builder.Services.AddOpenApi();

// 2. BUILD (creates the pipeline)
var app = builder.Build();

// 3. CONFIGURE middleware (order matters — runs top to bottom)
app.MapOpenApi();
app.UseHttpsRedirection();
app.UseAuthorization();
app.MapControllers();

// 4. RUN (start listening for HTTP requests)
app.Run();
LineWhat It DoesNestJS Equivalent
WebApplication.CreateBuilder(args)Creates the builder that sets up DI, config, loggingnew NestFactory.create(AppModule)
builder.Services.AddControllers()Registers all [ApiController] classesModule's controllers: [UserController]
builder.Services.AddOpenApi()Enables OpenAPI/Swagger doc generationSwaggerModule.setup()
builder.Services.AddInfrastructure()Our custom extension — registers EF Core, etc.imports: [InfrastructureModule]
app.UseAuthorization()Adds auth middleware to the pipelineapp.use(cookieParser()) / Guards
app.MapControllers()Maps routes from controller attributesapp.use(router)
Key concept: NestJS uses decorators + modules. ASP.NET Core uses middleware pipeline. Every request flows through the pipeline in order: OpenAPI → HTTPS redirect → Auth → Controller. This is like NestJS's app.use() chain but built into the framework.

The Five Projects

Open OmniBlox.sln in your IDE. You'll see:

ProjectRoleDepends On
OmniBlox.Domain Entities, Enums, Value Objects — pure business logic Shared
OmniBlox.Application CQRS Commands/Queries, Handlers, DTOs, Pipeline Behaviors Domain, Shared
OmniBlox.Infrastructure EF Core DbContext, JWT Service, Email, Caching Application, Shared
OmniBlox.Api Controllers, Middleware, Program.cs — the HTTP entry point Application, Infrastructure, Shared
OmniBlox.Shared Cross-cutting: Exceptions, Results, Extensions Nothing

Dependency Direction (The Golden Rule)

    Api
     ↓
  Infrastructure → Application → Domain → Shared
                                      ↓
                                   (nothing)

Dependencies always point inward. Domain knows nothing about databases, HTTP, or JWT. Application knows about Domain but not about PostgreSQL or email providers. This is the Dependency Inversion Principle — the "D" in SOLID.
