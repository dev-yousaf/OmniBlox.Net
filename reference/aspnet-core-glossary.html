Reference · ASP.NET Core Glossary & Clean Architecture Cheat Sheet

ASP.NET Core Glossary

Terms you'll encounter throughout the OmniBlox migration, with NestJS equivalents where applicable. Print-friendly for desk reference.

Architecture

Clean Architecture
Organizes code into layers with strict dependency direction: outer layers depend on inner layers, never vice versa. Inner layers know nothing about databases, HTTP, or UI.
↳ NestJS equivalent: Module separation via @Module(), but enforced at compile time via project references.
Dependency Inversion Principle (DIP)
The "D" in SOLID. High-level modules should not depend on low-level modules — both should depend on abstractions. In practice: Application defines interfaces, Infrastructure implements them.
↳ NestJS equivalent: @Injectable() with interfaces — same concept, same reason.
CQRS (Command Query Responsibility Segregation)
Separates writes (Commands) from reads (Queries). Commands mutate state, Queries return data — never both in one handler.
↳ NestJS equivalent: Manual service methods split (createProduct() vs getProducts()), but without the formal Command/Query classes.
MediatR
Library that implements the Mediator pattern in .NET. An IRequest<TResponse> (Command or Query) is sent via IMediator.Send() and routed to its matching IRequestHandler<TRequest, TResponse>.
↳ NestJS equivalent: A service method call, but MediatR decouples the caller from the handler — the controller never sees the handler class directly.
Pipeline Behaviors
MediatR middleware that runs on every Command/Query. Behaviors wrap the handler in a chain: logging → validation → tenant injection → audit logging → handler.
↳ NestJS equivalent: Guards + Interceptors combined into a single pipeline.
Vertical Slices
Organizing code by feature rather than by layer. In OmniBlox, each domain (Products, Sales, etc.) has its own folder containing Commands, Queries, Handlers, and DTOs — all in one place.
↳ NestJS equivalent: Each module folder with its controller, service, and DTOs.

ASP.NET Core Framework

Program.cs
The application entry point. Configures services (DI), middleware pipeline, and hosts the app. Equivalent to NestJS main.ts + AppModule combined.
Controller
Handles HTTP requests and returns responses. Decorated with [ApiController] and [Route("…")]. Methods map to HTTP verbs via [HttpGet], [HttpPost], etc.
↳ NestJS equivalent: @Controller() — nearly identical concept.
Dependency Injection (DI)
Built into ASP.NET Core via IServiceCollection. Services are registered in Program.cs or extension methods and constructor-injected. Three lifetimes: AddSingleton(), AddScoped() (per request), AddTransient() (per injection).
↳ NestJS equivalent: @Injectable({ scope: Scope.DEFAULT }) — but with AddScoped as the .NET default.
Middleware
Code that runs on every HTTP request/response as part of the pipeline. Examples: exception handling, authentication, logging, CORS. Order matters — they run in registration order.
↳ NestJS equivalent: app.use() global middleware + Guards + Interceptors.
[Authorize]
Attribute that requires authentication. Can specify roles: [Authorize(Roles = "OWNER,ADMIN")]. Can be applied at controller or action level.
↳ NestJS equivalent: @UseGuards(AuthGuard) + @Roles() decorator.
[AllowAnonymous]
Overrides [Authorize] on specific actions. Used for login, signup, password reset endpoints.
↳ NestJS equivalent: @AllowAnonymous() decorator.

Entity Framework Core

DbContext
Main class for database interaction. Contains DbSet<T> properties (one per entity). Registered with AddDbContext<T>() in DI.
↳ NestJS equivalent: PrismaClient.
DbSet<T>
Represents a database table. Querying DbSet<Product> generates SELECT * FROM Products. Adding to it stages an INSERT.
↳ NestJS equivalent: prisma.product.findMany() / prisma.product.create().
Fluent API (IEntityTypeConfiguration<T>)
Separate class per entity that configures table name, columns, keys, indexes, and relationships. Better than data annotations because it keeps entity classes clean of infrastructure concerns.
↳ NestJS equivalent: Prisma schema attributes: @id, @unique, @relation.
Migrations
Code-generated files that track database schema changes. Created with dotnet ef migrations add <Name>, applied with dotnet ef database update.
↳ NestJS equivalent: prisma migrate dev.
Global Query Filters
EF Core feature for multi-tenancy. A filter like builder.HasQueryFilter(e => e.CompanyId == tenantId) is automatically applied to every query. You never manually add .Where(e => e.CompanyId == id). No NestJS/Prisma equivalent — this is manual in the TypeScript codebase.
Navigation Properties
Object references between entities, e.g. User.Company (parent) and Company.Users (collection). EF Core uses these to generate JOINs.
↳ NestJS equivalent: Prisma relation fields like company Company @relation(...).
